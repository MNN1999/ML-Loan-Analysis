Loan Approval Policy Modeling + Anomaly Detection

This project takes a portfolio of ~20k loan applications and uses a simple, interpretable model to learn the bank's implicit loan approval policy. Once that policy is learned, the same model is used to scan the portfolio and flag loans where the historical decision does not match the behaviour suggested by the data.

In practical terms: historical approvals & rejections → learned approval policy → a shortlist of high-value anomalies for risk and underwriting teams to review.

Problem

The bank processes high volumes of loan applications, but the effective approval policy is implicit (distributed across underwriters and guidelines). The main inconsistency sits in the Medium-risk segment: similar applicants sometimes receive different outcomes.

Result: limited visibility into where decisions are consistent vs noisy, and no simple way to pinpoint off-policy approvals/rejections for review.

üìä

Analysis

We build an end-to-end, reproducible pipeline:
CSV → PostgreSQL (Neon) → SQL feature engineering (risk/income/amount/duration bands + affordability ratios) → Python EDA → Logistic Regression.

The model is trained to replicate historical approval decisions, then validated by risk and affordability segments to ensure probabilities are usable for rule-like thresholds.

‚úÖ

Solution

Use the trained model as a policy mirror:
Historical decision ↔ Predicted approval probability.

Focus on Medium-risk and flag high-confidence exceptions:
p ≥ 0.90 & Rejected → potential under-approval
p ≤ 0.10 & Approved → potential over-approval

Output: a short, high-signal review queue for underwriting QA, policy tuning, and training.

Data & pipeline ‚Äì technical architecture

The dataset is a synthetic but bank-like portfolio of ~20,000 loan applications. Each record contains customer characteristics, risk and behavioural signals, loan economics and the final decision LoanApproved.

The goal of the pipeline is simple: take this raw portfolio and build a clean, reproducible flow that goes from raw CSV → SQL → engineered features → model → anomaly flags.

1

Data layer

  • Source: synthetic CSV with ~20k loans.
  • Database: PostgreSQL on Neon (loan_data table).
  • Access: Python via SQLAlchemy.

Storage → reliable, queryable single source of truth.

2

Logic layer (SQL + Python)

  • SQL: banding logic and payment-to-income calculation.
  • Python: feature selection, EDA and model building.
  • scikit-learn Pipeline for preprocessing + logistic regression.

Raw values → business-aligned features → interpretable model.

3

Output & presentation layer

  • Anomaly flags and exception types stored in the final DataFrame.
  • Plots exported to figures/ for reuse in GitHub Pages and slides.
  • Notebook + this site tell one consistent storyline.

Model outputs → visual story → review-ready insights.

End-to-end pipeline (step-by-step)

üì•

1. Raw data in Python

Load the loan-level CSV into pandas and sanity-check: column types, ranges, missing values and basic statistics. At this stage nothing is modeled; we only make sure the input is shaped correctly.

Output: a clean DataFrame that matches the expected schema.

üóÑÔ∏è

2. Persist to PostgreSQL on Neon

Use SQLAlchemy to upload the raw DataFrame to a managed Postgres instance on Neon. The database becomes the single source of truth. Jupyter is now just a client that queries it.

Output: a loan_data table in the database.

üßÆ

3. Feature engineering in SQL

Directly in SQL, we derive:

  • Risk bands: Low / Medium / High based on RiskScore.
  • Income bands: <30k, 30‚Äì60k, 60‚Äì100k, 100k+.
  • Loan amount bands: <5k, 5‚Äì20k, 20‚Äì50k, 50k+.
  • Duration bands: ≤12m, 13‚Äì36m, 37‚Äì60m, 60m+.
  • Payment-to-income ratio: monthly payment / monthly income.

This step aligns the data with how a risk manager thinks in segments rather than raw numbers.

üêç

4. Modelling dataset in Python

We query the enriched table back into pandas and build model_df:

  • All relevant numeric features (RiskScore, CreditScore, DTI, total DTI, income, amount, duration, behaviour).
  • The engineered categorical bands (risk, income, amount, duration).
  • The target LoanApproved.

Result: one tidy dataset used for both EDA and ML.

ü§ñ

5. Logistic regression policy model

In scikit-learn, we build a Pipeline:

  • Numeric features → StandardScaler.
  • Categorical bands → OneHotEncoder.
  • Model → LogisticRegression(max_iter=1000).

The model is trained to predict LoanApproved and is evaluated on a held-out test set (accuracy, ROC-AUC, confusion matrix and per-class metrics).

üìê

6. Policy mirror & anomaly flags

Using predict_proba, we obtain an approval probability for every loan. These probabilities are:

  • Grouped into bands (<10%, 10‚Äì90%, >90%) to check calibration.
  • Compared against the actual decisions in the Medium-risk segment.

High-confidence disagreements between model and history are tagged as anomalies, with separate flags for “should probably have been approved” and “should probably have been rejected”.

7. Visual outputs & story

Finally, we produce visuals that summarise the entire pipeline: policy heatmaps, affordability boxplots, calibration bars and exception counts. These plots are exported as PNGs and reused in this page and in the notebook.

The visuals are the “front-end” of the project: they carry the story for business and risk stakeholders.

Model & insights

The core model is a logistic regression trained to mimic the bank's approval decisions. The power of the project lies in how this model is interpreted and used: first to understand policy, then to spot anomalies.

1. Policy heatmap: risk vs income

What we did & what it shows

  • How it's coded: in model_df, we group by risk_band and income_band, compute the mean of LoanApproved (i.e. approval rate), and unstack to a matrix. That matrix is passed into a heatmap.
  • What the output shows: Low-risk customers are almost always approved across income levels, High-risk customers almost never. Medium-risk customers show a gradient by income band.
  • How to read the visual: each cell is the share of applications approved in that segment. Darker/lighter colours show where the bank is more generous or conservative. This is the “top-level fingerprint” of the approval policy.
  • Why it matters: it anchors conversations with stakeholders: “Is this the risk appetite you expected across income groups?”

2. Affordability lens: Medium-risk PTI

How affordability behaves in practice

  • How it's coded: we filter to Medium-risk customers (risk_band == "Medium"), then plot the distribution of payment_to_income_ratio split by LoanApproved = 0/1 using a boxplot.
  • What the output shows: approved Medium-risk customers tend to have lower payment-to-income ratios than rejected ones, but there is overlap. Some rejected cases look affordable; some approved cases sit at relatively heavier burdens.
  • How to read the visual: median lines, quartiles and tails show how strict or relaxed the bank is in applying affordability rules to Medium-risk customers.
  • Why it matters: this is exactly the area where policy tends to drift over time and where anomaly detection later adds value.

3. Model calibration by probability band

Can we trust the model probabilities?

  • How it's coded: after training, we compute pred_approval_prob from clf.predict_proba(X). We then bucket these probabilities into three bands: <10%, 10‚Äì90%, >90%. For each band, we calculate the actual approval rate (LoanApproved mean).
  • What the output shows: the low band has almost no approvals, the high band has almost all approvals, and the middle band sits in between. This means the model probabilities are well aligned with historical decisions.
  • How to read the visual: each bar is a probability segment; the height is the true approval rate in that segment. We want “low ‚âà 0%” and “high ‚âà 100%”.
  • Why it matters: once calibration looks reasonable, using the model as a “policy mirror” (and setting high-confidence thresholds like <=10%, >=90%) becomes defensible.

4. Exception patterns: high-confidence anomalies

Where model and history disagree

  • How it's coded: we focus on Medium-risk customers and create flags: high_conf_under_approved for cases where pred_approval_prob >= 0.9 but LoanApproved == 0, and high_conf_over_approved for pred_approval_prob <= 0.1 but LoanApproved == 1. We then count each type.
  • What the output shows: there are very few of these high-confidence disagreements. That's good: the model mostly agrees with history, and the anomaly list is short.
  • How to read the visual: each bar is an exception type: “model says approve, history says reject” vs “model says reject, history says approve”. The height shows how many loans fall into each bucket.
  • Why it matters: these are the cases that deserve human attention: either the model highlights customers that may have been unnecessarily rejected, or approvals that are off-pattern given the bank's usual behaviour.

Reflection & Extensions

This work shows how an interpretable model can reconstruct the bank‚Äôs implicit approval policy and isolate a small set of high-confidence exceptions for review.

Next steps

  • Link to outcomes: connect decisions to realised defaults/loss to validate signal and impact.
  • Cost-based thresholds: tune approval/review cut-offs using expected loss and business value.
  • Operationalise QA: run as a monitored control with drift checks and a defined review workflow.

With real outcome data, the same framework can prioritise cases by financial impact and run as a scheduled control with monitoring, escalation rules, and feedback into policy and training.
